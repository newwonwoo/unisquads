# addr-refine 주소정제 파이프라인 — 절차 명세서 (코드 전수조사 기반)

작성일: 2026-07-16 · 대상: unisquads/public/app.js + api/*.py
방식: 실제 코드를 함수 단위로 전수조사하여 절차 추출 (추측 없음)

---

## 0. 전체 개요

```
원본주소 1건
  → [1차 정제] preprocess → cascade(L1~L3) → resolve → 상태판정 + 교차검증
  → [2차 정제] CONFIRMED만 → IROS 등기조회 → 등기고유번호
```

- 1차 정제 = 주소 → PNU (juso/네이버)
- 2차 정제 = PNU/주소 → 등기고유번호 (IROS)
- 핵심 원칙: 각 단계가 다음 단계의 안전장치. 무료 API(juso+네이버)만 사용.

## 1. 전처리 (preprocess) — app.js:382

원본 문자열을 정규화하고, 검색어(searchText)와 동/호(unit)를 분리한다.

### 1-1. 문자 정규화 (순서대로)
1. `toHalfWidth` — 전각 → 반각 (１２３ → 123)
2. `hanjaToHangul` — 한자 → 한글
3. `[,·.]` → 공백
4. `숫자 의 숫자` → `숫자-숫자` (123의4 → 123-4)
5. 괄호류 `()[]{}` → 공백
6. 한글·영문·숫자·공백·하이픈·슬래시 외 문자 → 공백
7. `stripSuffixes` — 불필요 접미 제거
8. `splitAdminPrefix` — 행정구역 접두 분리
9. `산12` → `산 12` (산 지번 띄우기)
10. `동143` → `동 143` (한글+숫자 붙은 것 띄우기)

### 1-2. 동/호 추출
- `extractUnit(s)` → dong, ho 분리
- `inferUnitFromNumbers` — 숫자 패턴에서 동/호 추론 (예: 101-401 → 101동 401호)

### 1-3. 지번 코어 추출 (extractJibunCore) — 핵심
지번 뒤 건물명·동호 잔존이 juso 실패의 최대 원인(실측 81%)이라, 지번까지만 남긴다.

**조건 분기:**
- **도로명주소면 건드리지 않음** (로/길 + 숫자 패턴 감지 시 skip)
- **지번을 찾았고 core에 동/리/읍/면/가 있으면** → core로 교체 (건물명·동호 절단)
- **지번은 없지만 core에 동/리/읍/면이 온전하면** → core로 교체 (상가·건물명 잔재 제거)
- **지번도 못 찾고 위 조건도 아니면** → 원본 검색어 유지 (건물명만 있는 주소 보호)

### 1-4. 반환
```
{ cleaned: 정규화 원본, searchText: 지번코어 검색어, unit: {dong, ho}, raw: 원본 }
```

### 1-5. extractJibunCore 내부 규칙 (app.js:292)
- 노이즈 선제거: 쉼표복수지번, 물결범위(~), 말미점, **외 필지(300외/300외3/300외2필지 → 300)**
- 토큰 순회하며:
  - 행정구역 토큰 → 유지 (건물명 만나기 전까지)
  - **다른 리 2개 연속** (영천리 반촌리) → 지번 앞 리 채택, 앞 리 제거
  - 같은 리 반복 → 중복 제거
  - 건물명 토큰(BUILDING_TOKEN) 만나면 → 이후 무시 (hitBuilding)
  - 숫자 토큰 → 지번으로 확정 (행정구역 직후일 때)

## 2. 캐스케이드 (cascade) — app.js:576

정제된 검색어로 주소 원천을 단계적으로 조회. **먼저 잡히면 멈춤.**

### L1 — juso 원문
```
juso(cleaned) 조회
→ 결과 있음 → fromJuso 후보 반환, level="L1"
```
원본이 정상이면 여기서 끝. (가장 신뢰)

### L2 — juso 재검색 (지번코어)
```
searchText ≠ cleaned 이고 juso(searchText) 결과 있음
→ fromJuso 후보 반환, level="L2"
```
노이즈 제거된 지번코어로 재시도.

### L3 — 네이버 지역검색 (구 L4, 카카오 대체)
**진입 조건: L1·L2 실패 + clients.naverLocal 존재 + 건물명 추출 성공**

카카오는 유료화(비즈월렛·결제카드 필요)로 제거됨. 네이버가 무료 + 옛주소 교정력 우위.

**검색어 3단계 (좁은 것부터):**
1. `시군구 + 읍면동 + 건물명` (예: 당진시 송악읍 반촌명지아파트)
2. `시군구 + 건물명`
3. `건물명` 단독

**캐싱:** 키 = `시군구|건물명|검색어` (건물명 단독 캐싱 금지 — 동명 타지역 방지)
- 고유 건물명당 1회만 호출 (실측 93% 절감)

**잡히면:**
```
top = naverItems[0]
naverAddr = roadAddress || address   (도로명 우선)
→ juso(modernizeSgg(naverAddr)) 재확정
   ├ juso 성공 → fromJuso (PNU 확보), naverPnuOk=true
   └ juso 실패 → fromNaver (PNU 미확보), naverPnuOk=false
→ level="L3", naverAddr, naverPnuOk 반환
```

**3단계 모두 0건:**
```
level="L4_NORESULT", humanInputError=true
```
(시스템오류와 구분됨 — API 장애는 throw로 전파)

### 최종 폴백 (아무것도 없음)
```
level=null, candidates=[]
```

## 3. 상태 판정 (refineAddress + resolve) — app.js:761

### 3-1. cascade 예외 처리
- cascade가 throw (네이버 API 장애 등) → **SYSTEM_ERROR** (재시도 대상)

### 3-2. L3(네이버) 결과 분기
- **naverPnuOk=true** → 아래 일반 CONFIRMED 경로로
- **naverPnuOk=false** → **NAVER_CONFIRMED_PNU_FAILED**
  - 네이버 주소는 정상, PNU만 미확보
  - IROS 진입 안 함 (CONFIRMED 아니므로)

### 3-3. 인적오류
- **humanInputError=true** → **HUMAN_INPUT_ERROR**
  - 네이버 3단계 정상호출했으나 전부 0건 = 주소 자체가 틀림

### 3-4. resolve (후보 → 상태) — app.js:720
```
dedupe(candidates).filter(c => c.admCd)   ← admCd 없는 후보 제거

- 0건 → FAILED (NOT_FOUND)
- 1건 → CONFIRMED (PNU 생성, irosQuery 생성)
- 2건+ → AMBIGUOUS (후보 목록 + 좁힘 레벨)
```

### 3-5. 교차검증 (CONFIRMED만) — app.js:816
```
validateRegion(cleaned, jibunAddr, sidoOnly = (level==="L3"))

- L3(네이버 확정): sidoOnly=true → 시도만 비교
  (네이버가 시군구/법정동 오류 교정한 것 허용: 부전타워 만화리→교리, 당진 군→시)
- L1/L2(juso/직접): sidoOnly=false → 시군구·법정동까지 비교

→ MISMATCH면 CONFIRMED → VALIDATION_FAILED로 강등
```

## 4. 상태값 전체 (7종)

| 상태 | 의미 | IROS 진입 |
|---|---|---|
| **CONFIRMED** | 주소 확정 + PNU 확보 | ✅ 예 |
| **NAVER_CONFIRMED_PNU_FAILED** | 네이버 주소 정상, PNU만 미확보 | ❌ 아니오 |
| **AMBIGUOUS** | 후보 2개+ (juso가 못 좁힘) | ❌ 아니오 |
| **VALIDATION_FAILED** | 지역 교차검증 불일치 | ❌ 아니오 |
| **HUMAN_INPUT_ERROR** | 네이버 정상조회 후 0건 (주소 오타) | ❌ 아니오 |
| **SYSTEM_ERROR** | API 장애 (재시도 대상) | ❌ 아니오 |
| **FAILED** | juso 0건 (미발견) | ❌ 아니오 |

## 6. 2차 정제 — IROS 등기조회

### 6-1. 진입 조건 — app.js:1615
```
status === "CONFIRMED" 만 등기조회 대상
```

### 6-2. IROS 검색어 (buildIrosQuery) — app.js:710
```
addr = jibunAddr || roadAddr || ""   (지번 우선, 없으면 도로명)
+ dong동 + ho호
```
- **지번주소 우선, 없으면 도로명주소로 폴백**
- 둘 다 없으면 빈 문자열 (단, CONFIRMED는 juso 거쳐 거의 항상 있음)

### 6-3. 배치 호출 — app.js:1648
```
addr = row.result.jibunAddr || row.result.irosQuery || ""
→ /api/resolve?addr={addr}&dong=&ho=&bdnm=
→ iros_api.resolve_one_api
```

### 6-4. IROS 내부 (iros_api.py)
- `_build_swrd` — 시도 뗀 지번주소 + 건물명 + 동호 조립
- 검색 순서: **지번만 먼저 → 안 되면 건물명 추가** (등기부 등록명이 juso와 다른 경우 대비)
- `_match_unit` — 동/호로 세대 특정 (선행0 int 비교: 0706=706)
- 단일동 건물: 동 없으면 호로만 매칭
- 부번 없는 대표지번: 271-8 실패 시 271 재시도
- 연결 끊김: 3회 재시도

### 6-5. IROS 결과 상태
RESOLVED / REG_MULTI / REG_NOT_FOUND / REG_SESSION_ERROR / REG_RATE_LIMIT / REG_PARSE_ERROR / REG_HTTP_ERROR / REG_TIMEOUT

## 7. 안전장치 요약 (다층 구조)

```
L1 juso원문 ─┐
L2 juso재검색 ├─ 원본/정제 지번을 juso가 아는가 (신뢰 최상)
             │
L3 네이버 ────┴─ 건물명으로 찾기 → juso 재확정(검증) → 시도 교차검증
                  ├ juso 검증 통과 + PNU → CONFIRMED → IROS
                  ├ juso 검증 실패 → NAVER_CONFIRMED_PNU_FAILED (IROS 보류)
                  └ 시도 불일치 → VALIDATION_FAILED (IROS 차단)

CONFIRMED만 IROS 진입 → 등기고유번호
```

**핵심:** 어느 단계도 단독 확정 안 함. 네이버(찾기)→juso(검증)→시도(안전) 3중.
정제 실패(FAILED)조차 "사람이 봐라" 신호. 채권관리 — 오확정 > 미확정 (틀린 확정이 더 나쁨).

## 8. 발견된 논점 (검토 필요 사항)

### 8-1. 건물명만 있는 원데이터의 흐름
```
건물명만 (지번·도로명 없음)
→ L3 네이버 검색 → 주소 확보
→ juso 재확정
   ├ 성공 → CONFIRMED → IROS ✅
   └ 실패 → NAVER_CONFIRMED_PNU_FAILED → IROS 진입 안 함
```
**논점:** juso 재확정 실패 시 네이버가 준 도로명주소로 IROS를 시도할 수 있으나(IROS는 도로명만으로도 검색 가능 — 실측), 현재는 IROS 진입을 막음. 이는 juso 검증을 안전장치로 두는 설계 의도. → **정책 결정 사항** (검증 우선 vs 커버리지)

### 8-2. IROS 검색어에 숫자가 전혀 없는 경우
- IROS는 "한글과 숫자만" 요구 = 숫자(지번/동번호)가 최소 하나 필요
- 도로명 + 동번호면 검색됨 (지번 불필요 — 실측)
- 숫자 전무(건물명만)는 빈 검색어 → IROS 거부. 단 CONFIRMED에선 거의 발생 안 함
