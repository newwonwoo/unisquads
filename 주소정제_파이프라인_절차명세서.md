# 주소정제·IROS 처리 절차 명세서

기준일: 2026-07-19
판정 기준: 주석이나 과거 문서가 아니라 `public/app.js`, `public/*.mjs`, `api/iros_api.py`의 실행 코드

## 1. 처리 단위와 핵심 원칙

주소정제는 행 단위 결과를 만들되, 배치에서만 알 수 있는 근거는 주소군 단위로 계산한다. IROS는 세대 행마다 호출하지 않고 PNU 단위로 후보를 완전 수집한 뒤 각 세대의 동·호를 로컬 매칭한다.

| 구간 | 처리 단위 | 자동 확정 조건 |
|---|---|---|
| 주소 전처리·검색 | 고유 원문주소 | 주소 후보와 지역 검증 통과 |
| 주소군 보강 | 우편번호·소유자·법정동 그룹 | 그룹 내 직접 확정 PNU가 정확히 1개 |
| IROS 후보 수집 | PNU | 원본 수신 수가 `totalRecordCount`와 정확히 같고 파싱 오류 0건 |
| IROS 세대 매칭 | PNU+동+호 | 완전 후보 전체에서 부동산구분·동·호·검토근거 통과 후 1건 |

오확정 방지가 우선이다. 부분수집, 파싱누락, 부동산구분 불일치, 검토근거 불충분에서는 `REG_UNIT_NOT_FOUND`나 `RESOLVED`를 만들지 않는다.

## 2. 가공소 구성

### 2.1 필수 가공소

모든 주소가 순서대로 거친다.

1. 문자 정규화: 전각, 한자, 공백, 괄호, 지번 표기 정리
2. 주소 구조화: 시도·시군구·읍면동·도로명·지번·건물명·동·호 추출
3. JUSO 검색: 도로명 또는 지번 구조로 현행 주소와 PNU 후보 확보
4. 후보 해소: PNU 중복 제거, 지번·건물명·용도 확인
5. 지역 검증: 시도·시군구·법정동 비교
6. 상태 확정: `CONFIRMED`, `AMBIGUOUS`, `VALIDATION_FAILED`, `FAILED` 등으로 분리

JUSO 주소 API는 주소와 PNU를 확정하는 기준 원천이다. `hstryYn=Y`로 변동주소도 검색한다.

### 2.2 선택 가공소

필요한 행만 통과하며 한 행이 여러 선택 가공소를 연속으로 거칠 수 있다.

| 선택 모듈 | 진입 조건 | 역할 |
|---|---|---|
| 네이버 복구 | JUSO 결과 없음 + 건물명 있음 | 건물명으로 지번/도로명 후보를 찾고 다시 JUSO로 PNU 확인 |
| 우편번호 복구 | 시군구 부족 + 6자리 구우편번호 | 단일 또는 인접 지역 후보를 순차 검색 |
| 주소군 힌트 | 동일 주소군에 반복 패턴 존재 | `N-M`이 지번인지 동·호인지 집단 판정 |
| 동소 복구 | 원문에 `동소` 존재 | 같은 행정구역 선행 기준행의 건물명으로 참조어 치환 |
| 옛주소 변환 | 명시된 과거 행정구역 토큰 존재 | 현행 토큰으로 재검색하고 규칙 ID를 결과에 기록 |
| 주소군 전파 | 식별주소가 없는 실패행 + 그룹 PNU 1개 | 주소/PNU만 전파하고 각 행 동·호와 원문은 보존 |

네이버는 최종 PNU 원천이 아니다. 네이버가 찾은 주소를 JUSO가 다시 확인해야 일반 `CONFIRMED`가 된다. API 장애는 0건과 구분해 `SYSTEM_ERROR` 또는 일시 오류로 남긴다.

## 3. 행정구역 검증

`public/admin-successor.mjs`가 허용 가능한 옛주소→현행주소 변환을 버전과 규칙 ID로 관리한다.

- 시군구 예: 마산시→창원시, 청원군→청주시, 당진군→당진시
- 읍면동 예: 배방면→배방읍, 용진면→용진읍, 고령읍→대가야읍, 대소면→대소읍, 동산동→여의동, 고강본동→고강동
- 방향은 과거→현행만 허용한다.
- 같은 어간이라는 이유만으로 임의의 `면→읍`, `리→동`을 통과시키지 않는다.
- 시군구가 달라도 법정동이 같다는 일반 예외는 제거했다. 지역 한정 개편은 시도 조건이 있는 명시 규칙만 허용한다.

사용한 규칙은 `validation.oldAddressMap`과 엑셀의 옛주소 진단 열에 남는다.

## 4. 배치 멱등성

각 업로드 행에는 원본 행과 분리 세대를 포함하는 결정적 `rowId`가 부여된다. 결과 객체는 행마다 복제하므로 중복주소 처리나 주소군 전파가 다른 행의 결과 객체를 함께 변경하지 않는다.

각 결과는 다음 메타데이터를 가진다.

- `pipelineVersion`
- 실제 적용된 `appliedModules`
- 적용 모듈별 `moduleVersions`
- 원문·우편번호·원본 부가열·동호 override·상위 근거의 `dependencyFingerprint`
- 결과 내용의 `resultFingerprint`

재실행 시 입력, 적용 모듈 버전, 주소군 힌트, 동소 기준, 전파 기준행/그룹 근거가 모두 같고 결과 자체가 변조되지 않은 행만 재사용한다. 일시 오류와 시스템 오류는 재사용하지 않는다. 사용하지 않은 선택 모듈의 버전 변경은 해당 행을 무조건 재처리시키지 않는다.

주소군 전파는 기준행 ID, 기준 PNU, 그룹 구성원, 기준행 결과 지문으로 `groupFingerprint`와 `evidenceHash`를 만든다. 기준행 PNU나 그룹 구성이 바뀌면 기존 전파 결과가 무효화되고 새 근거로 갱신된다.

## 5. IROS 단건과 배치 전략

### 5.1 단건

지번주소와 입력 동·호·건물명을 전달한다. 정확검색으로 충분한 응답이면 해당 응답을 판정에 사용하고, 완전성이 확인되지 않으면 부재를 확정하지 않는다.

### 5.2 배치

1. `CONFIRMED`만 대상으로 한다.
2. 집합건물인데 호가 없으면 호출하지 않고 `UNIT_INPUT_REQUIRED`로 분리한다.
3. PNU가 같은 행을 한 그룹으로 묶는다.
4. 그룹 첫 행의 지번주소로 `strategy=full`을 한 번 호출한다.
5. 후보를 최대 `pageUnit=1000`으로 순차 수집한다.
6. 완전 후보 캐시를 같은 PNU의 모든 세대가 공유한다.
7. 각 행의 동·호로 후보를 로컬 매칭한다.

`pageUnit=1000`은 한 페이지 상한이지 1,000건 초과 포기 기준이 아니다. 서버 응답이 허용하면 다음 페이지를 같은 세션에서 계속 수집한다. 45초 안전예산 안에 끝낼 수 없으면 `REG_COLLECTION_DEFERRED`로 보류한다.

## 6. IROS 완전성 계약

완전 캐시는 아래를 모두 만족해야 한다.

```text
complete == true
query_scope == EXACT_LOT
raw_received_count == total_count
parse_error_count == 0
content_hash 존재
schema_fingerprint 존재
collector/parser 버전 일치
```

캐시에는 총건수, 원본 수신수, 파싱수, 고유후보수, 페이지수, 유효 pageUnit, 조회시각, 수집기·파서·매처 버전, 응답 내용 해시와 스키마 지문을 저장한다. 완전성 조건을 못 채운 응답은 PNU 완전 캐시로 저장하지 않는다.

세션별 pageUnit 수용 능력은 메모리 캐시에 기록해 같은 실행 중 불필요한 상한 탐색을 반복하지 않는다.

## 7. 세대 매칭 계약

1. 폐쇄 등기를 제외하고 현행을 우선한다.
2. `real_cls_cd` 또는 부동산구분으로 집합건물 여부를 확인한다.
3. 동·호 정규화 시 알파벳 동과 `204-1` 같은 하이픈 호를 보존한다.
4. 후보 동·호는 파서가 `buld_no_room → buld_no_inner → 상세문구` 순서로 보강한 값을 사용한다.
5. 전체 후보에 동이 없고 호만 있는 단일동 건물은 호만으로 재매칭한다.
6. 주소정제 결과에 검토 플래그가 있으면 지번 범위, 집합건물 구분, 동·호에 더해 건물명까지 일치해야 한다.
7. 한 건이면 `RESOLVED`, 여러 건이면 `REG_MULTI`다.
8. `REG_UNIT_NOT_FOUND`는 완전수집과 파싱완료가 증명된 후보 전체를 확인한 뒤에만 가능하다.

부분응답은 `REG_PARTIAL_RESPONSE`, 파싱누락은 `REG_PARSE_INCOMPLETE`, 수집예산 초과는 `REG_COLLECTION_DEFERRED`, 부동산구분/검토근거 실패는 `REG_VALIDATION_FAILED`로 분리한다.

## 8. 엑셀 감사 정보

결과 엑셀은 원본 열과 업무 결과 외에 다음을 제공한다.

- 주소상태와 세대상태 분리
- 파이프라인/결과/의존성 지문과 적용 모듈
- 옛주소 규칙 ID·버전·입력·현행 토큰
- 주소군 전파 기준행과 근거 해시
- IROS 전략·검색범위·부동산구분
- 총건수·원본수신수·파싱수·고유후보수·페이지수·유효 pageUnit·완전여부
- IROS 파서/매처 버전, 후보 캐시 해시, 동·호 매칭 근거

이 열들로 “왜 확정됐는지”, “없음 판정이 완전 후보에 근거했는지”, “재실행에서 왜 다시 처리됐는지”를 역추적할 수 있다.

## 9. 현재 버전

| 구성 | 버전 |
|---|---|
| 주소 파이프라인 | `addr-pipeline-v1` |
| 옛주소 맵 | `admin-successor-v1` |
| IROS 수집기 | `iros-collector-v4` |
| IROS 파서 | `iros-parser-v3` |

버전 문자열은 캐시와 재사용 판정의 계약값이므로 동작을 바꾸면 함께 갱신한다.
